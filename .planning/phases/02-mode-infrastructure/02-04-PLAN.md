The mode toggle button is currently inside the footer `<div>` at the bottom of the sidebar. That div has the class `collapsed ? "opacity-0 h-0 p-0 overflow-hidden" : "opacity-100"`, which completely hides the button when the sidebar is collapsed on desktop. This makes the toggle completely inaccessible in collapsed state, violating MODE-01 ("visible from any page").

**The fix:** Remove the toggle button from the footer div and place it at the END of the `<nav>` block, after the advanced items section (`{isAdvanced && (...)}` block) and before the closing `</nav>` tag. This mirrors how other nav items behave: icon visible when collapsed, icon+label when expanded.

**Surgical diff — what to change:**

Step 1: Remove the toggle Button from the footer div (lines ~203–220 in the current file). The footer div should only contain the `{t("sidebar.footer")}` text after this change:

```tsx
{/* Footer */}
<div className={cn(
  "p-4 border-t border-border transition-all duration-200",
  collapsed ? "opacity-0 h-0 p-0 overflow-hidden" : "opacity-100"
)}>
  <div className="text-xs text-muted-foreground text-center">
    {t("sidebar.footer")}
  </div>
</div>
```

Step 2: Add the mode toggle button inside `<nav>`, after the `{isAdvanced && (...)}` block and before `</nav>`. Add a `<Separator className="my-2" />` above it to visually separate it from the nav items (whether simple or advanced mode):

```tsx
        {/* Mode toggle — always visible, even when collapsed (icon-only) */}
        <Separator className="my-2" />
        <Button
          variant="ghost"
          size="sm"
          disabled={isUpdating}
          onClick={() => setMode(isAdvanced ? 'simple' : 'advanced')}
          className={cn(
            "w-full justify-start gap-3 h-10 transition-all duration-200",
            collapsed && "justify-center px-0 gap-0"
          )}
        >
          <BarChart3 className="h-4 w-4 flex-shrink-0 text-muted-foreground" />
          <span className={cn(
            "transition-all duration-200 font-medium text-sm",
            collapsed && "opacity-0 w-0 overflow-hidden"
          )}>
            {isAdvanced ? t('mode.switchToSimple') : t('mode.switchToAdvanced')}
          </span>
        </Button>
```

The `<nav>` tag already has `overflow-y-auto` — this is fine; the toggle will scroll with nav content only if there are many items. For the typical 6–9 items, it will always be visible without scrolling.

**Do not change:** the existing sidebarItems map, the advancedItems map, the footer div's existence (keep it — just without the toggle), the collapsed behavior of nav items, any imports (BarChart3 is already imported), or the header collapse button.

Phase 2 success criterion #2 currently reads:

> 2. After refreshing the browser, the app opens in the same mode the user last used — no flash to wrong mode

This criterion is not satisfied as implemented: ModeContext initializes with `mode='simple'` and syncs asynchronously from DB via useEffect. Advanced-mode users see simple UI briefly on cold load before `setModeState('advanced')` fires. The decision to not add an `isLoading` guard was made explicitly in [02-01]: "No isLoading guard in AdvancedRoute — cold-cache flash deferred until it becomes a real user issue."

The flash is safe-directional: it goes from simple to advanced (never shows advanced content to a simple-mode user). The DB sync is correct and mode persists correctly. The criterion language overstates the guarantee.

**Update the success criterion in .planning/ROADMAP.md:**

Old:
```
  2. After refreshing the browser, the app opens in the same mode the user last used — no flash to wrong mode

A brief flash from simple to advanced on cold cache load is an accepted trade-off (safe-directional: no advanced content is shown to simple-mode users). An isLoading guard was explicitly deferred per [02-01] decision log.
