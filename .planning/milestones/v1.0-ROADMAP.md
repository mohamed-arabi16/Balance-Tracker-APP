# Roadmap: Balance Tracker — Advanced Mode

## Overview

This milestone adds a toggleable freelancer "Advanced Mode" on top of the existing personal finance tracker. The build follows a strict dependency order: database schema first (nothing else is possible without types), then mode infrastructure (the gate that controls access), then clients (the central entity all invoices depend on), then transaction-client linking (enriches client data before invoices arrive), then the full invoice lifecycle with PDF export, and finally the advanced analytics dashboard that aggregates everything. Each phase delivers a coherent, independently verifiable capability. Simple mode is never regressed — all advanced features are purely additive.

## Phase Details

### Phase 1: Database & Type Foundation
**Goal**: All new tables, columns, and RLS policies exist in Supabase; TypeScript types reflect the full schema; the app compiles against the new types with no regressions
**Depends on**: Nothing (first phase)
**Requirements**: None (foundational — unlocks all other phases)
**Success Criteria** (what must be TRUE):
  1. Supabase dashboard shows new tables: clients, invoices, invoice_items; new columns: incomes.client_id, expenses.client_id, user_settings.app_mode
  2. Every new table has RLS enabled and a user-scoped SELECT/INSERT/UPDATE/DELETE policy — verified in Supabase Auth > Policies
  3. src/integrations/supabase/types.ts reflects the new schema and the app builds with zero TypeScript errors
  4. Deleting a client does not cascade-delete income or expense rows (SET NULL behavior confirmed via manual test)
  5. Two concurrent invoice creations for the same user produce unique, sequential invoice numbers (no duplicates)
**Plans**: 3 plans

Plans:
- [x] 01-01-PLAN.md — Write SQL migration file (all 7 schema changes in FK-dependency order)
- [x] 01-02-PLAN.md — Apply migration to Supabase and verify cascade behaviors (human checkpoint)
- [x] 01-03-PLAN.md — Update TypeScript types, DEFAULT_USER_SETTINGS, and confirm zero typecheck errors

### Phase 2: Mode Infrastructure
**Goal**: Users can switch between Simple and Advanced mode, the preference persists across sessions, and Advanced routes are inaccessible while in Simple mode
**Depends on**: Phase 1
**Requirements**: MODE-01, MODE-02, MODE-03, MODE-04
**Success Criteria** (what must be TRUE):
  1. A mode toggle button is visible from any page; clicking it switches the app between Simple and Advanced mode with no page reload required
  2. After refreshing the browser, the app opens in the user's last saved mode — mode persists correctly via DB sync. A brief flash from simple to advanced on cold cache load is an accepted trade-off (safe-directional: no advanced content is shown to simple-mode users). An isLoading guard was explicitly deferred per [02-01] decision log.
  3. Navigating directly to /clients, /invoices, or /advanced while in Simple mode redirects the user to the dashboard
  4. User can open Settings and select a default mode; that preference is saved and used on next login from a new device

- [x] 02-02-PLAN.md — App.tsx provider chain + /advanced route, AdvancedDashboard stub, i18n keys

### Phase 4: Transaction-Client Linking
**Goal**: When creating income or expense entries, users in Advanced mode can optionally tag a client — existing Simple mode entry flows remain identical for users who do not use Advanced mode
**Depends on**: Phase 3
**Requirements**: TXN-01, TXN-02
**Success Criteria** (what must be TRUE):

  2. When in Advanced mode, the expense entry form shows an optional "Client" dropdown with the same behavior as income
  3. When in Simple mode, income and expense forms show no client field — the Simple mode experience is identical to before this phase

### Phase 5: Invoices & PDF Export
**Goal**: Users in Advanced mode can create invoices with line items, manage invoice status through its full lifecycle, export any invoice as a PDF, and all UI strings have Arabic translations matching app quality standards
**Depends on**: Phase 3
**Requirements**: INV-01, INV-02, INV-03, INV-04, INV-05, INV-06, I18N-01
**Success Criteria** (what must be TRUE):
  1. User can create an invoice linked to a client with one or more line items (description, quantity, rate); the invoice is assigned a unique sequential number automatically on creation
  2. User can transition an invoice from Draft to Sent to Paid; an invoice with a past due date and status "Sent" is automatically shown as Overdue without manual intervention
  3. A Draft invoice can be fully edited; a Sent or Paid invoice displays as read-only with no edit controls
  4. Clicking "Export PDF" on any invoice downloads a PDF file containing the invoice details; the PDF library is not loaded until the user clicks the button (main bundle stays within size budget)
  5. When marking an invoice as Paid, the app shows a prompt offering to create a matching income entry; user can confirm or dismiss — no income record is created unless confirmed
  6. All Advanced Mode UI strings (invoices, clients, dashboard) display correctly in Arabic with proper RTL layout when the app language is set to Arabic

### Phase 6: Advanced Dashboard
**Goal**: Users in Advanced mode have a dedicated dashboard at /advanced showing revenue per client and outstanding invoices — giving a complete financial picture of their freelance business at a glance
**Depends on**: Phase 5
**Requirements**: DASH-01, DASH-02
